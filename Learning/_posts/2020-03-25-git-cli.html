Distributed Version Control System

Git is a distributed version control system, which is different from central version control system.

Central VCS refers to a single remote central database. Problem is if something wrong happens to the central database then none of the remote device would be able to connect to it. If the database is corrupted then we'd better hope it's backed up somehow.

For distributed VCS, everybody has a local copy of the repository. You can connect to the remote repo, however the local has everything that the remote had based on the last time they are synced.

Basically in a Distributed VCS, every developer has a back-up of the central repository.

Installing & Setup

After installation, do this to check if everything is OK.

git --version

This indicates that git is installed OK:

git version 2.17.1

Seting up global configuration variables:

git config --global user.name "Kenny Nguyen"
git config --global user.email "kkenny.nguyen@gmail.com"

After this we can do

git config --list

which will check all global config variables.

If you need help:

git help config
git config --help

The format for help is either git help <verb> or git <verb> --help.

Getting Started

Two common scenarios:

1. Initialize a Repo from existing code

We have an existing project on our local machine and we want to start gitting.

Preparation

We need to do:

git init

after we cd to the local repo. What it would do is iniatializing an empty Git repo in the directory with the address local-repo/.git/.

.git is a directory that contains everything that's related to our repository.

Before first commit:

git status

.gitignore

If we have files that contains personal data or something that is not relevant to our code, we can do

touch .gitignore

.gitignore file is just a simple text file that we can add files that we want git to ignore. We can use wildcards.

Add files to staging area

Staging area is the area that we organize what we want to commit to Git.

[WORKING DIRECTORY] —prepare changes—> [STAGING AREA] —commit—> [git]

git add -A

to add eveyrthing .. or we can just add files individually.

remove from staging area:

git reset filename

or just git reset which would just return everything from staging area to working directory.

Commit

After we git add -A to add everything to Staging area, we can do git commit -m "Initial Commit".

After this if we do git status, it will show "working directory clean". This means we've committed the files and we have no modified or untracked files.

We can also do git log that will give us the Hash number of our commit.

2. Tracking an existing remote project

Cloning a remote repo

git clone <url> .

The . means to clone everything.

$ git remove -v: List information of the repository.
$ git branch -a: List all branches of the repository.

Pushing changes

$ git diff: show the changes that we've made to the codebase.
$ git status: show the modified file
$ git add -A: add to staging area
$ git status again: file is ready to be commited
then..

$ git commit -m "abcdef"

then

$ git pull origin master: pull any changes that have been made while we are working. This is useful for code bases that have a lot of people working on.
$ git push origin master: push to master branch of remote origin repo.

origin refers to the name of the remote repository.
master refers to the branch we want to push/ pull to.

Common workflow

Instead of working on the master branch, we shouldn't be doing that. We should be creating a branch for the desired feature by:

git branch abc-def

and then if we do git branch without the branch name, it will list all the branches. The one with asterisk (*)next to it is the one we are currently working on.

To start working on the branch we just created abc-def, we need to do:

git checkout abc-def

Then if we rerun $ git branch again, abc-def will have an asterisk next to it.

Steps to undertake are:

STEPS EFFECTS
$ git init Initiate a new repo
$ git clone URL clone repo
   
$ git branch abc-def Creating a local branch
$ git checkout abc-def Start working on the branch
Make changes Changes are made to the Working Dir
$ git add -A Add changes to Staging Area.
$ git commit -m "I am making a change. Changes are commited to local branch. No effect on master branch and/or remote repo.
$ git push -u origin abc-def The -u option associates local abc-def branch to remote abc-def branch. After that we don't need to do that anymore

WORKING DIRECTORY > INDEX > HEAD
Holds the actual files and folder. (git add) Staging Area (git commit) Points to the last commit

Merging a branch to master

Branches are used to develop features isolated from each other. The master branch is the default branch when creating a repo. Use other branches for development and merge them back to the master branch upon completion.

$ git checkout master

to start working on master branch.
then..
$ git pull origin master

then we can do $ git branch --merged to see which branch has been merged with master.
then..
$ git merge branch-name

then..
$ git push origin master to push the changes to the master branch to remote repo origin.

ALTERNATIVELY: Instead of pushing directly to master branch, we can work on a different branch (let's call it feature_x):

git checkout feature_x
git push origin feature_x

.. then, to delete the local branch, we can do $ git branch -d branch-name to delete the merged branch by the -d option.
This would delete the local branch.

To delete a remote branch do this $ git push --delete branch-name.

A branch will not be available to other unless it has been pushed to the remote repo using $ git push origin branch-name.

Differences between git remote add vs git clone:

Example 1:

git clone https://github.com/abc

The above creates a new git local repository by copying an existing remote one located at the URL.

Example 2:

git remote add test https://github.com/abc

The above does not create any local git copy. It only creates a pointer for the name of https://github.com/abc, which in the above equal test.

Differences between git pull and git fetch

A git pull is basically a git fetch + merge.

  • git fetch: connects to remote repo (origin) and fetches new commits.
  • git merge: incoporates new commits into local branch.
git pull == git fetch origin && git merge origin

Git Workflow

A Git workflow is a recipe or a recommendation, or a "standard process" for how to use Git to accomplish work in a consistent, productive manner. It encourages users to leverage Git effectively and consistently.

Git offers a lot of flexibility in how users manage changes. Give its focus on flexibility, there is no "standardized process" on how to interact with Git.

When working with a team on a Git managed project, it is important to make sure that everyone is in agreement on how the flow of changes will be applied.

There are several publicized Git workflows that can be a good fit for a team. Read more in here: Comparing Workflows.
