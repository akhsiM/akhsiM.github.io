Concurrency

General

Concurrency is when the execution of two or more pieces of code act as if they run at the same time. This is different to Parallelism, which is when they actually run at the same time.

To have concurrency, the code needs to run in an environment that can switch execution between different parts of the code when it's running. This is often implemented using things such as fibers, threads, and processes.

Concurrency is about software whereas parallelism is more about hardware. To have parallelism we need actual hardware that can do two things at once. This might be multiple cores in a CPU, multiple CPUs in a computer or multiple computers connected together.

Concurrency is a requirement if we want our application to be practical in the real world, where almost everything happens asynchronously. If the whole of our process is serial with one thing happening before moving on to the next, our system would feel sluggish and we wouldnt be taking full advantage of the hardware our program runs on.

Concurrency in Python

Without concurrency

In Python, code can be run concurrently using the threading module.

In the below program we run the do_something() function once. All this does is sleep for one second.

We time this function. As expected, the script takes one second to run. We run the function once, and it takes one second.

Any time a program runs synchronously, it actually does not do much on the CPU and is just waiting around like the above. These are good opportunities to implement threading and concurrencies.

There are two different concepts: CPU-bound and IO-bound tasks. CPU-bound tasks are programs that are compute-heavy and use a lot of CPU. IO-Bound tasks are tasks that are not using the CPU that much but are simply waiting for input/ output. Example include read/ write from file system, network operation, downloading..etc..

Threading is most suited for IO-bound tasks. These are when our programs do a lot of waiting around for input/ output operation.

If a task is CPU-bound, we need to use parallelism and not threading. As a matter of fact, CPU-bound tasks would actually degrade in performance if we implement threading due to the overhead addition of starting and killing a thread.

Running codes concurrently doesn't mean that they are run at the same time. Concurrency gives the illusion of running the code at the same time because instead of waiting for i/o, our program would simply move on to the next function.

Let's try threading with Python. We don't have to run pip install because threading comes with Python.

Instead of running the do_something() functions synchronously, we'll turn them both into threads. In the target= bit, we are not going to be inserting the () bit because we aren't executing the function but only to pass it in un-executed.

Passing the function as a target of a thread does not actually run it:

import threading

def do_something():
        print('Sleeping for 1 second...')
        time.sleep(1)
        print('Done Sleeping..')

start = time.perf_counter()

t1 = threading.Thread(target=do_something)
t2 = threading.Thread(target=do_something)

finish = time.perf_counter()

print(f'Finished in {round(finish-start,2)} second(s)')

Finished in 0.0 second(s)

Instead, we'll need to run the threads as well. We need to use the start method on each thread.

In the above example, we are missing the Done Sleeping.. bit. Why? It's because our script continued on and finished while the threads were still sleeping.

We need to wait for our threads to finish before finishing the script. This can be done with the join method.

import time
import threading

def do_something():
        print('Sleeping for 1 second...')
        time.sleep(1)
        print('Done Sleeping..')

start = time.perf_counter()

t1 = threading.Thread(target=do_something)
t2 = threading.Thread(target=do_something)

t1.start()
t2.start()

t1.join()
t2.join()

finish = time.perf_counter()

print(f'Finished in {round(finish-start,2)} second(s)')

Sleeping for 1 second...
Sleeping for 1 second...
Done Sleeping..
Done Sleeping..
Finished in 1.0 second(s)

This is good now. Our script actually waited for the two threads to re-join before finishing. As above, our code runs the do_something() function twice and finishes in one second. Without concurrency it took us two seconds.

With concurrency - the easier way

Let's replace the manual way of implementing concurrency with a different, easier and more efficient way.

This is actually not in the threading module but in the concurrent.futures module. When we use this, we should also be using a context manager.

Below, we'll submit the schedule for the execution of do_something to be run. We can also pass in an argument to the submit method. After we run the submit method, the f1 object is a "future" object which encapsulates the execution of function do_something().

We can also "check-in" on the result of the future object, assuming the it "returns" something. This is achievable with the result() method. For this we need to change print('Done Sleeping..') to return 'Done Sleeping..'.

  • with the result() method:
    import time
import concurrent.futures

def do_something(seconds):
        print(f'Sleeping for {seconds} second(s)...')
        time.sleep(seconds)

        # In order to allow us to check in on the function, we'll return the result instead of printing it.
        return 'Done Sleeping..'

start = time.perf_counter()

with concurrent.futures.ThreadPoolExecutor() as executor:
        f1 = executor.submit(do_something, 1.5) 
        # The result method waits around until the function completes.
        print(f1.result())

finish = time.perf_counter()

print(f'Finished in {round(finish-start,2)} second(s)')

    Sleeping for 1.5 second(s)...
Done Sleeping..
Finished in 1.5 second(s)

  • Note about Exception

    If our function raises an Exception, it won't be raised while running the thread. The Exception will be raised when the result is iterated.

Multiprocessing

Multiprocessing is about running codes in parallel.

Why would we want to use multiprocessing? We would use multiprocessing to speed up our program. The speeding-up comes from different tasks running in parallel.

Let's take an example with a function that simply takes a one-second sleep.

import time

start = time.perf_counter()

def do_something():
        print('Sleeping 1 second...')
        time.sleep(1)
        print('Done sleeping..')

do_something()

finish = time.perf_counter()

print(f'Finished in {round(finish-start,2)} second(s)')

Sleeping 1 second...
Done sleeping..
Finished in 1.0 second(s)

If we instead run this function twice, it would expectedly take 2 second for the program to finish.

import time

start = time.perf_counter()

def do_something():
        print('Sleeping 1 second...')
        time.sleep(1)
        print('Done sleeping..')

do_something()
do_something()

What we were doing is running the tasks synchronously or in serial like this:


In concurrency, tasks that are sped up are IO-bound tasks like opening a resource, downloading a file, ..etc..

In multiprocessing, we can speed things up by spreading work out to multiple processors on our machine and run the tasks at the same time. This is applicable to both IO-bound and CPU-bound tasks.

Multiprocessing looks like this:


Multiprocessing in Python - the old way

To do this in our previous script:

import time
import multiprocessing

start = time.perf_counter()

def do_something():
        print('Sleeping 1 second...')
        time.sleep(1)
        print('Done sleeping..')

p1 = multiprocessing.Process(target=do_something)
p2 = multiprocessing.Process(target=do_something)

p1.start()
p2.start()

# Make sure the process is finished and not asleep.
p1.join()
p2.join()


finish = time.perf_counter()

print(f'Finished in {round(finish-start,2)} second(s)')

Multiprocessing in Python - the newer way

We can implement multiprocessing using a new tools. This is a process pool executor.

Similarly to concurrency, the library that we need to import is concurrent.futures. It's best to use this library within a context manager.

In the example below, the f1 variable is turned into a "future" object that encapsulates the scheduled processing of the function in the future.

We can also check-in on the result of the function.

Using mapping for arguments

What map does is run the function with every item in the list secs. When use the list of argument, the executor returns a list of future object. When we use map, Python returns just the results.

import time, concurrent

start = time.perf_counter()

def do_something(seconds):
        print(f'Sleeping {seconds} second(s)...')
        time.sleep(seconds)
        return f'Done sleeping for {seconds}s..'

with concurrent.futures.ProcessPoolExecutor() as executor:
        secs = [5, 4, 3, 2, 1]
        processes = executor.map(do_something, secs)
        
        for result in processes:
