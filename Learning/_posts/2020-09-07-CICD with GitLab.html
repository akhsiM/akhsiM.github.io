Introduction to CI/CD

What is CI/CD ?

Continuous Delivery is not magic. It is about continous improvements at all levels of organizations. It is about embracing the discipline of constantly pursuing higher performance and excellence by accelerating feedback loops.

In software development, CI/CD refers to the combined practices of continuous integration and either deployment or delivery. CI/CD bridges the gaps between development and operation activities and teams by enforcing automation in building, testing and deployment of applications.

CI/CD is a set of principles, patterns and practices designed to make software deployments predictable and routine affairs that can be performed on demand at any time.

These methodologies are based on using scripts for automation to minimize the chance of introducing human errors while developing applications. Essentially, with a CI/CD process, none to minimal human intervention is required from the point of new code development until its deployment.

Codes are continuously built, tested and deployed at every small iteration. This reduces the chance of developing new code based on a bugged or failed previous version.

Let's say, you and another developer are working on a project. It is better for each of you to regularly merge your work and fixing issues as they come up, rather than to work separately for months and then find out bugs later down the line.

There are three main approaches to this methodology, each of them can be applied to suit our own strategy.

Continuous Integration


The ELI5 version of CI is "as soon as my code is commited, the server does something with it". This something can include compiling, running tests and even deploying our application without us having to lift a finger. Aside from minimizing human intervention, we also have an instant feedback loop, created through the building and upholding the culture of continuous actions of check-ins and builds.

To implement CI, along with introducing automated building and testing into our workflow, we also need to start with the mindset. Continuous Integration makes no sense when development teams don't take alerts seriously. Arming developrs with the tools is only half of the equation. Adopting CI as a practice starts with changing the mindset of you and your colleague. It takes the whole team commitment and understanding the value of addressing issues immediately to improve the quality and sustainability of your team collective development effort. Developers should merge/ commit their changes back to their main branch as often as possible.

The practice of Continuous Integration is to create a set of scripts that can be used to automate the process of building and testing our application. These will run automatically whenever we push some codes into our repository. These continuous actions decrease the chances of introducing errors to our app. With these processes automated, developers should also be encouraged to commit more regularly into the shared repository. This way, if there is any errors, we can catch it then and there.

Achieving "true CI/CD" means:

  • Daily or regular commits to main branch
  • Automated triggering of build and test
  • Repair within minutes of failure

What we need (cost):

  • We need to write automated tests for each new feature, improvement or bug fix.
  • We need a continuous integration server that monitors the main repo and run scripts automatically for every new commits pushed.
  • Developers need to merge changes as often as possible. Ideally once a day..

What we gain (benefits):

  • Less bugs are captured in production because regressions are captured early by the automated tests.
  • Building release is easy because all integration issues have been solved early.
  • Less context switching for developers as they are alerted as soon as possible as they break the build.
    • Example, Kenny works on a Chat function, Kenny pushes a code commit to the main repo and it failed to build straight away, Kenny then immediately tries to address the issue before moving to another task.
  • Reduced testing costs as CI servers can run hunderds of tests in seconds.
  • QA team spend less time testing and can focus on improvements to the quality culture.

Continuous Delivery

Continuous Delivery is an extension of continuous integration to make sure that we can release changes to our customers quickly in a sustainable way.

If CI stops at automated building and testing, Continuous Delivery adds automated release process into the equation. In theory, with Continuous Delivery we can decide the frequency of our release ourselves to suit the business requirements. With this, we are encouraged to deploy to production as early as possible to make sure that releases are done in small batches, which makes issues easier to troubleshoot.

The point of Continuous Delivery is to make software release deployment painless and low-risk events that can be performed at anytime on demand.

This practice ensures the code is checked automatically but ultimately still requires human intervention to manually and strategically trigger the deployment of the changes.

What we need (cost):

  • Strong foundation of continuous integration, e.g our test suite needs to cover our codebase.
  • Deployments need to be automated. The trigger to deploy is still manual, however when the deployment process is started, there should be no need for a human to intervene.
  • Team may need to embrace "feature flags" so that incomplete features do not affect customers in production.

What we gain (benefits):

  • The complexity of deploying software is taken away. Our team does not have to spend a day preparing for the release any more.
  • We can release more often, accelerating the feedback loop with customers.
  • Much less pressure on decisions for small changes, encouraging faster iteration (Agile principles).

Continuous Deployment


Continuous Deployment continues where Continuous Integration and Continuous Delivery left off. It introduces automated deployment into the picture.

With this practice, every changes that passes all stages of our production pipeline is released to the customers. There is no human interaction, only a failed test will prevent new change to be deployed to production.

What is the point of Continous Deployment? It further accelerates the feedback loop between software developers and customers and take pressure off the team as there is no "Release Day" anymore.

What we need (cost):

  • Our testing culture needs to be at its best. Our release quality will depend on the quality of our test suite.
  • Our documentation process also needs to keep up with the pace of deployments.
  • Feature flags become an important part of the release process to make sure we can coordinate with other departments.

What we gain (benefits):

  • We can develop faster.
  • Releases are less risky and easier to fix in case of problems.
  • Customers can see a continuous stream of improvements and enhancements. Quality increases every day instead of say, every quarter.

GitLab CI/CD

GitLab CI/CD is a powerful tool built into GitLab that allows us to apply all continuous methods to our software without third-party application or integration needed.

Init Ci/CD

In order to use GitLab CI/CD, all we need is an application codebase hosted in a Git repository and for our build, test, and deployment scripts to be specified in a file called .gitlab-ci.yml located in the root path of our respository.

The .gitlab-ci.yml file

In this .gitlab-ci.yml configuration file we can define:

  • The scripts we want to run
  • Include and cache dependencies
  • How our commands should run, in parallel or in sequence
  • Where our app is deployed to
  • Our scripts to be run automatically, or triggered, or manually

More advanced steps can be added into this file as we become more familiar with it.

Once a .gitlab-ci.yml file is added to the root of our repository, GitLab will detect it and run our scripts with the tool called GitLab Runner which works similarly to a terminal.

The scripts are grouped into jobs and together they compose a pipeline.

Here is a minimal example of .gitlab-ci.yml:

before_script:
	- apt-get install rubygems ruby-dev -y

run-test:
	script:
		- ruby --version

The before_script attribute would install the dependencies that our app needs before running everything. Then, a job called run-test would print the Ruby version of the current system. Both of these jobs compose a pipeline that is triggered at every push to any branch of the repo.
